\chapter{\texorpdfstring{Vulnerability 5.46b: Tainted\\Length (Packet OOB Write)}{Vulnerability 5.46b: Tainted Length (Packet OOB Write)}}

\section{Executive Summary}

Vulnerability 5.46b is a \textbf{confirmed memory corruption vulnerability} where an attacker-controlled \textbf{tainted packet field (TCP Data Offset)} is used directly as a loop counter in a memory write operation, causing a \textbf{buffer overflow within the in-kernel packet buffer}. The corruption writes to adjacent \textbf{packet-local memory} (Ethernet header fields), not kernel data structures.

\begin{tcolorbox}[colback=blue!5!white,colframe=blue!75!black,title=ISO Rule Violated,fonttitle=\bfseries]
ISO-IEC Rule 5.46: ``Do not apply the sizeof operator to a polymorphic object'' (and related: tainted values in sinks)
\end{tcolorbox}

\begin{tcolorbox}[colback=red!5!white,colframe=red!75!black,title=Severity,fonttitle=\bfseries]
\textbf{MEDIUM:} Direct memory corruption confirmed, but \textbf{scope-limited to packet-buffer memory}. Overwrites adjacent packet fields (MAC addresses), not kernel structures. \textbf{No privilege escalation possible}.
\end{tcolorbox}

\subsection{Why This Vulnerability Matters}

\begin{enumerate}[label=\arabic*.]
  \item \textbf{Tainted Packet Field}: TCP Data Offset (doff) is attacker-controlled
  \item \textbf{No Validation}: Value is used directly without bounds checking
  \item \textbf{Fixed-Size Buffer}: Destination MAC address buffer is only 6 bytes
  \item \textbf{Verifier Gap}: Doesn't enforce field-level bounds checking within packet structures
  \item \textbf{Scope Limitation}: Overflow stays within packet-local memory, not kernel memory
\end{enumerate}

\section{The Vulnerable Code Pattern}

\subsection{Vulnerable Code (Simplified)}

\vspace{0.3cm}
\noindent
\fcolorbox{black!30}{gray!15}{%
  \begin{minipage}[c]{\dimexpr\textwidth-2\fboxsep-2\fboxrule\relax}
    {\small\ttfamily
    /* Fixed-size buffer for MAC destination (6 bytes standard) */\\
    unsigned char h\_dest[6];\\
    \\
    /* Get tainted length from packet TCP header */\\
    \_\_u32 tainted\_length = hdr-\textgreater{}tcp-\textgreater{}doff * 4;\\
    /* doff is 4-15 (4-bit field), so length is 20-60 bytes */\\
    bpf\_printk("[taintsink]: Tainted length \%u", tainted\_length);\\
    \\
    /* Use tainted length in a memory write loop (NO VALIDATION!) */\\
    for (i = 0; i < tainted\_length; i++) \{\\
    \phantom{for (i = 0; i < tainted\_length; i++) \{}h\_dest[i] = /* packet\_data[i] */;\\
    /* BUFFER OVERFLOW if tainted\_length > 6 */\\
    \}
    }
  \end{minipage}%
}
\vspace{0.3cm}

\subsection{Why This Is Vulnerable}

\begin{enumerate}[label=\arabic*.]
  \item \textbf{Fixed-size buffer}: \texttt{h\_dest} is \textbf{only 6 bytes} (standard Ethernet MAC address)
  \item \textbf{Tainted length}: \texttt{doff * 4} is attacker-controlled via TCP Data Offset field
  \item \textbf{Range}: TCP doff is a 4-bit field (values 4-15), so \texttt{doff * 4} = 20-60 bytes
  \item \textbf{No validation}: Code directly uses tainted value as loop counter
  \item \textbf{Overflow}: Writing 60 bytes to a 6-byte buffer = \textbf{54-byte overflow} into adjacent memory
\end{enumerate}

\section{Exploitation Analysis}

Memory corruption has been confirmed: the verifier accepts the program, the bytecode preserves the 60-byte loop, the xlated bytecode has no bounds guards, runtime executes all 60 writes, the attacker controls the length (via TCP data offset), and adjacent memory is corrupted with the MAC source address overwritten. Packet memory layout shows that the \texttt{h\_dest} buffer is 6 bytes (indices 0-5), \texttt{h\_source} is 6 bytes (indices 6-11), and \texttt{h\_proto} is 2 bytes (indices 12-13). With doff=15 (60-byte write), bytes 0-5 are written to \texttt{h\_dest} (intended), bytes 6-11 overflow to \texttt{h\_source} (first overflow), bytes 12-13 overflow to \texttt{h\_proto} (second overflow), and bytes 14-59 write beyond the packet header (third overflow). The impact is scope-limited: writes stay within the packet buffer structure (not kernel stack), the overflow cannot reach kernel memory, the attacker cannot modify task credentials or eBPF context, and while the packet may be corrupted, network processing continues with limited DoS potential. While direct exploitation for kernel compromise is impossible due to eBPF sandbox constraints---execution is isolated from kernel memory, the eBPF stack is separate from kernel stack, and corrupted data affects only packet-local memory, not kernel control flow---this vulnerability demonstrates a critical verifier gap: the verifier doesn't enforce field-level bounds and only checks packet-level safety, representing a systematic gap in bounds checking that affects all packet buffer field access patterns. The vulnerability is exploitable within packet-buffer scope, but the eBPF sandbox prevents kernel-level compromise.

\subsection{Packet Construction: Crafting the Overflow}

\noindent
To trigger the vulnerability, the attacker must construct a TCP packet where:

\begin{enumerate}[label=\arabic*.]
  \item The total IP packet length accommodates a large payload
  \item The TCP Data Offset (\texttt{doff}) field indicates 60+ bytes
  \item The packet contains attacker-controlled data in those bytes
\end{enumerate}

\subsection{Verifier's Taint Analysis Failure}

\noindent
The verifier's taint analysis fails because:

\begin{enumerate}[label=\arabic*.]
  \item The payload length is tainted (user-controlled from packet)
  \item The verifier checks that the packet is at least 60 bytes (memory safety)
  \item However, it does not check that the 6-byte field is not overflowed
  \item The verifier conflates packet-level bounds with structure-level bounds
\end{enumerate}

\noindent
Result: The verifier accepts a bounds check that is sufficient for packet memory safety but insufficient for structure-field safety. Detailed verifier analysis is available in the evidence logs.

\noindent
\textbf{Impact:} Overwrites the next 54 bytes in the packet buffer, potentially corrupting subsequent packet header fields or other eBPF map data.

The verifier's bounds check confirms the packet has at least 60 bytes, but this does not constrain writes to the 6-byte structure field. Detailed bytecode analysis is available in the evidence logs.

\subsection{Verifier's Taint Analysis Failure}

\noindent
The verifier needs to:
\begin{enumerate}[label=\arabic*.]
  \item Mark the payload length as tainted (from packet)
  \item Propagate the taint through all operations using it
  \item Require explicit validation (bounds check) before using tainted values
  \item Verify that the bounds check is sufficient (check against the actual buffer size)
\end{enumerate}

\noindent
If the verifier accepts a bounds check that is not tight, tainted values can escape and cause OOB writes.

\section{Classification: EXPLOITABLE (BUT SCOPE-LIMITED)}

\noindent
The verifier accepts the program, the bytecode contains the overflow, runtime executes the overflow, the attacker controls overflow data (packet payload) and overflow length (packet doff field), enabling direct writes to \textbf{packet-local buffer memory} with attacker-controlled data. \textbf{This is a confirmed memory-corruption vulnerability, but corruption is limited to packet-buffer scope, not kernel-level structures}. No privilege escalation or kernel compromise is possible because the overflow targets in-kernel packet buffer fields, not kernel stack or task credentials.

\section{Scope of Impact: Packet-Buffer-Limited}

\noindent
The vulnerability demonstrates a verifier bounds-checking gap, but the impact is \textbf{constrained}:

\subsection{What CAN be corrupted}

\begin{enumerate}[label=\arabic*.]
  \item Adjacent Ethernet header fields (MAC source: hdr->eth->h\_source)
  \item Packet protocol field
  \item Other in-flight packet metadata within the same buffer structure
\end{enumerate}

\subsection{What CANNOT be corrupted}

\begin{enumerate}[label=\arabic*.]
  \item Kernel stack memory (the overflow is within packet buffer)
  \item Saved return addresses (not accessible from packet buffer)
  \item Task credentials (not adjacent to packet fields)
  \item Kernel function pointers (not within packet memory)
\end{enumerate}

\subsection{Why Privilege Escalation is Impossible}

\begin{enumerate}[label=\arabic*.]
  \item The overflow is confined to \textbf{packet-local memory} within the eBPF context
  \item Packet buffers are temporary, stack-allocated data structures in XDP handler
  \item Modifying packet fields does not affect kernel task credentials or return addresses
  \item The eBPF sandbox prevents escape to general kernel memory
\end{enumerate}

\section{Constraints and Realities}

\noindent
\begin{itemize}
  \item \textbf{Reliability}: Stack values are predictable in kernel context
  \item \textbf{Return address location}: Depends on XDP processing context; may require guessing or information disclosure
  \item \textbf{Control flow}: Returning from the overflow point must land in a useful code path
\end{itemize}
