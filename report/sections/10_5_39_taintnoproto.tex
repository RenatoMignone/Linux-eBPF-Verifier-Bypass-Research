\chapter{\texorpdfstring{Vulnerability 5.39: Unprototyped\\Function Pointer (Stack OOB Write)}{Vulnerability 5.39: Unprototyped Function Pointer (Stack OOB Write)}}

\section{Executive Summary}

Vulnerability 5.39 is a \textbf{confirmed memory corruption vulnerability} combining an unprototyped function pointer (ISO-IEC Rule 5.39 violation) with a \textbf{tainted array index derived from the TCP sequence number}, enabling \textbf{out-of-bounds writes to the eBPF-local stack frame}. This is directly exploitable but \textbf{sandbox-limited} — corruption cannot escape the eBPF context to compromise the kernel.

\begin{tcolorbox}[colback=blue!5!white,colframe=blue!75!black,title=ISO Rule Violated,fonttitle=\bfseries]
ISO-IEC Rule 5.39: ``Do not create pointers to functions with an implicit return type''
\end{tcolorbox}

\begin{tcolorbox}[colback=red!5!white,colframe=red!75!black,title=Severity,fonttitle=\bfseries]
\textbf{MEDIUM-HIGH:} Direct memory corruption confirmed. Out-of-bounds stack writes with attacker-controlled offset and value. DoS-capable. \textbf{Privilege escalation impossible due to eBPF sandbox isolation}.
\end{tcolorbox}

\subsection{Why This Vulnerability Matters}

\begin{enumerate}[label=\arabic*.]
  \item \textbf{Unprototyped Function Pointer}: Verifier loses tracking of what the function does and what arguments it accepts
  \item \textbf{Tainted Array Index}: TCP sequence number is attacker-controlled and used directly as an array index
  \item \textbf{No Bounds Checking}: Verifier cannot prove the index stays within bounds [0-15]
  \item \textbf{Direct Memory Write}: Can corrupt adjacent eBPF-local stack memory
  \item \textbf{Sandbox Constraint}: eBPF execution is isolated; corruption cannot reach kernel memory or task credentials
\end{enumerate}

    \phantom{static \_\_always\_inline void restricted\_sink(int i) \{} bpf\_printk("index : \%d", i);\\
    \phantom{static \_\_always\_inline void restricted\_sink(int i) \{} s.array1[i] = 42;  /* OUT-OF-BOUNDS if i >= 16 */\\
    \}\\
    \\
    /* Unprototyped function pointer */\\
    void (*pf)() = restricted\_sink;  /* No prototype! */\\
    \\
    /* Tainted input from TCP packet */\\
    \_\_u32 seq\_value = bpf\_htons(hdr-\textgreater{}tcp-\textgreater{}seq) / 1000;\\
    (*pf)(seq\_value);  /* Verifier loses range info at call boundary */
    }
  \end{minipage}%
}
\vspace{0.3cm}

\subsection{Why This Is Vulnerable}

\begin{enumerate}[label=\arabic*.]
  \item \textbf{Unprototyped pointer}: `void (*pf)()` — No parameter type info, verifier can't track argument constraints
  \item \textbf{Tainted input}: TCP sequence number is attacker-controlled (values 0-$2^{32}-1$)
  \item \textbf{Division doesn't constrain}: `seq / 1000` is still unbounded (0-4,294,967)
  \item \textbf{Array size is 16}: Accessing array1[16+] writes beyond buffer into adjacent stack memory
  \item \textbf{Verifier failure}: Without knowing the expected range [0-15], verifier can't prevent OOB access
\end{enumerate}

\textbf{Interpretation}:
- R3 is loaded from packet (line 139-141)
- No bounds check on R3
- Store instruction uses R3 as an offset (line 149)

\textbf{Finding}: Bytecode shows the unprototyped call with tainted index. Not optimized away.

\section{Exploitation Analysis}

Memory corruption has been confirmed: the verifier accepts the program, the bytecode preserves the vulnerability, the xlated bytecode has no guards, runtime executes out-of-bounds writes, the attacker controls the index (via TCP sequence number), and the attacker controls the write value (fixed at 0x42). The `array1[16]` buffer is only 16 bytes (indices 0-15 valid), so writing to indices 16-25 overwrites 10 bytes of adjacent memory into `array2[0-9]`, modifying adjacent eBPF-local variables. For example, index 25 with value 0x42 writes to offset 25, corrupting 10 bytes into adjacent memory. The vulnerability can crash the XDP handler via corrupted stack frame, cause kernel panic if corruption affects critical control structures, or disable network processing on the interface, and it is reliable because the attacker controls the index value (TCP sequence number), any SYN packet can trigger it, and multiple packets can trigger multiple OOB writes. However, privilege escalation is impossible due to eBPF sandbox constraints: execution is isolated from kernel memory, the eBPF stack is separate from kernel stack, and corrupted return addresses affect only the eBPF execution context (not kernel), so memory corruption is \textbf{eBPF-local only} and not kernel-level compromise.
