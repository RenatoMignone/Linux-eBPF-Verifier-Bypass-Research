\chapter{\texorpdfstring{Vulnerability 5.39: Unprototyped\\Function Pointer (Stack OOB Write)}{Vulnerability 5.39: Unprototyped Function Pointer (Stack OOB Write)}}

\section{Executive Summary}

As detailed in Chapter 6, vulnerability 5.39 combines an unprototyped function pointer (ISO-IEC Rule 5.39 violation) with a tainted array index derived from the TCP sequence number, resulting in stack out-of-bounds write capability. This section documents the practical bytecode evidence, verifier analysis, and runtime exploitation confirming the vulnerability's severity.

\begin{tcolorbox}[colback=blue!5!white,colframe=blue!75!black,title=Key Finding,fonttitle=\bfseries]
An attacker can use the TCP sequence number as a direct array index to write attacker-controlled values to arbitrary stack offsets, enabling return address overwrites or critical data corruption. This is a memory corruption vulnerability (EXPLOITABLE).
\end{tcolorbox}

\section{Source-Level Patch}

The vulnerability is introduced via a Git patch located at:

\begin{small}\raggedright
\texttt{XDPs/xdp\_synproxy/patches/5\_39\_taintnoproto/}\\
\texttt{0001-feat-5.39-taintnoproto\_2-Tainted-potentially-}\\
\texttt{mutilated.patch}
\end{small}

\noindent
Applied to \texttt{xdp\_synproxy\_kern.c}. This patch adds an unprototyped function pointer and tainted array access within a TCP sequence number-derived index.

\noindent
The vulnerability combines:
\begin{enumerate}[label=\arabic*.]
  \item An unprototyped function pointer call (verifier cannot validate its behavior)
  \item A TCP sequence number (attacker-controlled via incoming SYN packet)
  \item A fixed-size stack array accessed with the sequence number as index
\end{enumerate}

\noindent
Simplified pseudocode:

\vspace{0.3cm}
\noindent
\fcolorbox{black!30}{gray!15}{%
  \begin{minipage}[c]{\dimexpr\textwidth-2\fboxsep-2\fboxrule}
    {\small\ttfamily
    // Unprototyped function pointer (missing return type and args)\\
    void (*unproto\_fn)();\\
    \\
    // Array on the stack (size 16 x u16 = 32 bytes)\\
    u16 stack\_array[16];\\
    \\
    // Inside tcp\_dissect(), tainted TCP sequence number\\
    u16 seq\_num = tcp\_hdr-\textgreater{}seq;  // Attacker controls this!\\
    \\
    // Call through unprototyped pointer with tainted value\\
    // The function uses seq\_num as an array index\\
    unproto\_fn(seq\_num);\\
    \\
    // Inside the function (unknown signature to verifier):\\
    void target\_fn(u16 idx) \{\\
    \phantom{void target\_fn(u16 idx) }stack\_array[idx] = 0x42;  // NO BOUNDS CHECK!\\
    \}\\
    }
  \end{minipage}%
}
\vspace{0.3cm}

\noindent
\subsection{Why This Is Dangerous}

\noindent
The TCP sequence number is a 32-bit value controlled by the attacker (the remote client sending the SYN packet). If the verifier cannot validate the bounds of \texttt{idx}, then index values such as 16 access \texttt{stack\_array[16]} out-of-bounds, while values such as 25 write 50 bytes beyond the array into adjacent stack memory containing return addresses, saved registers, and other local variables.

This proves the compiler preserved the unsafe data flow. Detailed bytecode analysis is available in the evidence logs.

\subsection{Packet Construction: Triggering Out-of-Bounds Indices}

\noindent
To trigger the vulnerability, we must craft TCP packets where the sequence number, after conversion and division, yields index values $\geq 16$.

\noindent
\textbf{Challenge:} The code performs:
\begin{enumerate}[label=\arabic*.]
  \item \texttt{bpf\_htons(seq)} — byte-order swap
  \item Division by 1000
  \item Implicit cast to array index (lower bits)
\end{enumerate}

\noindent
\textbf{Solution:} To target index 16, we need \texttt{(bpf\_htons(seq) / 1000) \& 0xFFFF = 16}, which yields \texttt{seq = 16000} in network byte order. This requires carefully positioning the desired value in the lower 16 bits after byte-order conversion.

\noindent
\textbf{Example Payload:} The PoC constructs packets with TCP sequence numbers set to values like \texttt{0x3E80...} (representing 16000 after processing) to reliably trigger out-of-bounds array accesses.

\subsection{Verifier Taint Tracking Failure}

\noindent
The verifier should:
\begin{enumerate}[label=\arabic*.]
  \item Recognize that \texttt{seq\_num} is tainted (user-controlled)
  \item Require explicit bounds checking before array access
  \item Reject the program if tainted value is used as an index without validation
\end{enumerate}

\noindent
However:
\begin{enumerate}[label=\arabic*.]
  \item The unprototyped function pointer call breaks the verifier's taint tracking
  \item The verifier cannot determine what the function does with its arguments
  \item The array access happens inside the called function, which is opaque to the verifier
  \item Result: The verifier accepts the program despite the tainted array access
\end{enumerate}

Detailed verifier analysis is available in the evidence logs.

\subsection{Memory Impact}

\noindent
Each write to \texttt{stack\_array[idx]} where \texttt{idx > 16} writes to kernel stack memory immediately adjacent to the array. Depending on stack layout, an index of 16 overwrites 32 bytes beyond the array end, while an index of 25 overwrites 50 bytes, potentially corrupting saved registers, other local variables, or return addresses depending on the memory layout.

\section{Classification: EXPLOITABLE}

\noindent
\begin{itemize}
  \item \checkmark Verifier accepts the program
  \item \checkmark Bytecode preserves the tainted index and array access
  \item \checkmark Runtime executes OOB writes
  \item \checkmark Memory corruption is confirmed (writes to stack beyond array bounds)
  \item \checkmark Attacker controls the index value (via TCP seq number)
  \item \checkmark Attacker controls the data being written (0x42)
\end{itemize}

\textbf{This is a confirmed memory-corruption vulnerability}.

\section{Exploitation Potential}

\noindent
\subsection{Immediate Threats}

\begin{enumerate}[label=\arabic*.]
  \item \textbf{Logic corruption}: Overwrite local variables that influence control flow
  \item \textbf{Pointer modification}: If adjacent memory contains a function pointer, overwrite it
  \item \textbf{Return address hijacking}: If the layout aligns, overwrite saved RIP (x86-64)
\end{enumerate}

\subsection{Constraints}

\begin{enumerate}[label=\arabic*.]
  \item Stack layout is fixed at function entry; precise overwrites require knowledge of stack frame layout
  \item The XDP context may not allow arbitrary shellcode execution (depends on subsequent execution context)
  \item Limited to a single write per packet (triggering the vulnerability once)
\end{enumerate}
